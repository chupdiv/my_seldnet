# Причины плохого обучения модели (SeldNet_3classes_80, fs=250 kHz, 4 мика)

Отчёт по анализу генератора датасета (`generate_reverb_all_inclusive_gpu_voice.py`) и пайплайна `SeldNet_3classes_80`, с учётом **внесённых правок в коде**.

Базовые параметры STFT (общие для большинства задач):

| Параметр | Значение |
|---|---|
| `fs` | 250 000 Гц |
| `hop_len_s` | 0.02 с → `hop_len = 5 000` сэмплов |
| `win_len` | 10 000 сэмплов |
| `nfft` | 16 384 |
| `label_hop_len_s` | 0.04 с |
| `feature_label_resolution` | 2 |
| `nb_mel_bins` | **256** (GCC+Mel; ранее 64 — см. ниже) |
| LPF генератора | 5 000 Гц |
| `mel_fmax_hz` | 5 000 Гц (согласовано с LPF) |
| Геометрия 4 микрофонов | квадрат в плоскости YZ, сторона **0.1 м** (рёбра ~0.1 м, диагональ ~0.141 м) |
| Макс. TDOA | 412 мкс ≈ **103 сэмпла** |

---

## Внесённые исправления в код (референс)

| Что сделано | Где |
|---|---|
| `nb_mel_bins = 256` — окно GCC ±128 лагов, покрывает TDOA до 103 сэмплов | `parameters.py` |
| `fmax_spectra_salsalite = 5000` — спектральная часть SALSA не тянет «пустую» полосу 5–9 кГц | `parameters.py` |
| `freq_bins_input` для SALSA считается динамически (как в `cls_feature_class`), не константа 382 | `cst_former/CST_details/cst_geometry.py` |
| `max_tau = 128` для задач 9, 10, 333; зажим TDOA в `[-max_tau, max_tau]` при построении таргета | `parameters.py`, `train_seldnet.py` (`TdoaLoss`) |
| Координаты микрофонов в `TdoaLoss` — те же декартовы точки, что `MIC_GEOMETRIES[4]` в генераторе (квадрат 0.1 м в YZ) | `train_seldnet.py` |

**Чекпоинты:** старый `models/9_tdoa-3tracks-16channels.h5`, обученный при `max_tau=6`, **не совпадает с размерностью** слоёв NGCC после увеличения `max_tau`. Для задач **10** и **333** нужен новый претрейн task **9** или обучение без совместимых весов.

---

## Оставшиеся и исторические причины проблем

### 1. GCC-PHAT: узкое окно при `nb_mel_bins = 64` (историческая критическая ошибка)

**Статус:** исправлено в конфиге (`nb_mel_bins = 256`).

При `nb_mel_bins = 64` окно `±32` лага не покрывало TDOA до 103 сэмплов — пики кросс-корреляции обрезались. После правки окно **±128** лагов достаточно.

После смены `nb_mel_bins` нужно **пересчитать фичи** (`batch_feature_extraction.py <task>`) для всех задач на GCC+Mel.

### 2. Несоответствие аудио и аннотаций в env_only/both (умеренная)

**Статус:** без изменений в генераторе.

Сохраняется полный клип, лейблы — только на активных интервалах; в паузах возможны хвосты реверберации при SED=0. Влияние умеренное; для `drone_only` не проявляется.

### 3. SALSA-Lite: «пустые» бины при `fmax_spectra_salsalite = 9000` (историческая)

**Статус:** исправлено в конфиге (`fmax_spectra_salsalite = 5000`).

При старом значении 9000 Гц при LPF 5 кГц значительная доля бинов приходилась на полосу без сигнала. Сейчас при `nfft=16384`, `fmin=50` Гц, `fmax_spectra=5000` Гц число бинов SALSA ≈ **324** (все в полезной полосе до среза).

### 4. SALSA-Lite: нецелое деление частоты при MaxPool (умеренная, остаётся)

Число SALSA-бинов (≈324) **не делится нацело** на произведение `f_pool_size`:

| Task | `f_pool_size` | Произведение | 324 / произведение |
|---|---|---|---|
| 5, 7 | [4, 4, 2] | 32 | 10.125 |
| 34 | [1, 4, 6] | 24 | 13.5 |

`MaxPool2d` обрезает хвост — часть частотной информации теряется на границе. Это не устранено правками, при необходимости можно подобрать `f_pool_size` или число бинов под целое деление.

### 5. CST-former + SALSA: хардкод `freq_bins_input = 382` (историческая)

**Статус:** исправлено — используется `freq_bins_input` из `cst_geometry.py` с динамическим расчётом для SALSA.

### 6. `max_tau = 6` при TDOA до 103 сэмплов (историческая)

**Статус:** исправлено в конфиге (`max_tau = 128`) и зажимом TDOA в `get_tdoa_target`.

### 7. Eigenmike в `TdoaLoss` вместо геометрии генератора (историческая)

**Статус:** исправлено — заданы декартовы координаты квадрата 0.1 м в YZ, как в генераторе.

### 8. Цепочка задач 9 → 10 → 333

**Статус:** логика зависимости сохраняется. После смены `max_tau` и геометрии нужен **новый** претрейн task 9 и обновлённый путь к весам для 10/333.

### 9. Задачи FOA (2, 3)

**Статус:** без изменений.

Код ожидает `foa_dev/` и амбисонику; генератор создаёт только `mic_dev/`. С текущим синтетическим датасетом эти задачи **не применимы**.
